Introduction

Motivation

CAD-based radition transport as a robust and practical analysis tool

  • equal freedom in deisgn and analysis
  • engineering analysis on the same model

CAD-Based Monte Carlo Radiation Transport

  • pathway for robust particle transport on CAD geometries exists
  • not yet fully practical:
    • some upstream CAD issues
    • long simulation times (2-20 times longer than native codes)

This can mean hours or days of additional simulation runtime

Research Goal

CAD-based radiation transport performance competitive with native Monte Carlo geometry representations
  • mult-faceted approach:
    • acceleration data structure construction improvements based on DAGMC workflow
    • extension of architecture-based accelerations used in rendering to raditaion transport
    • addition of acceleration data structure adapted from dynamic implicit surface rendering

Background

MC Geometry Representations

Native Geometry CAD Geometry
  • variants of Computational Solid Geometry (CSG)
  • volumes formed from boolean combinations of simple implicit surfaces
  • higher order surface complexity
  • convenient design tools
  • extrude, sweep, loft, splines, etc.

DAGMC

transport performed on CAD-based surface meshes

Why?

  • geometry queries can be difficult/impossible to satisfy analytically
    • higher-order root finding
  • surfaces are discretized into planar objects (triangles)
  • generalized algorithms and spatial searches for triangle intersections

MC Geometry Queries

Three common queries:

  1. volume point containment
  2. next surface intersection distance
  3. surface crossing angle

Ray Tracing

  • field dedicated to these types of queries on discretized surfaces
  • accomplished via spatial acceleration data structures in O(logN)
  • combined with geometry-based mesh topology for a particle tracking algorithm

  • KD-Tree

    • efficient traversal algorithms
    • large memory footprint

  • Octree

    • large memory footprint

  • Bounding Volume Hierarchies

    • relatively low memory footprint
    • fast to build
    • currently favored

Bounding Volume Hierarchies

bounding sphere

  • poor object bounding
  • simple intersection check
  • low memory storage (center and radius)

axis-aligned bounding box (AABB)

  • good object bounding
  • more complex intersection-check (6-planes)
  • more memory required (6 extents)

oriented bounding box (OBB)

  • tightly bounds objects
  • higher complexity of intersection-check (ray coord. transformation)
  • higher memory storage (AABB plus axes orientation)
